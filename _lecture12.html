Multi-Tenancy and Elasticity

Multi-Tenancy

  • Architectural approach where resources are shared between multiple tenants or consumers
  • Implications
    • Centralization of infrastructure in locations with lower costs
    • Peak-load capacity increases
    • Utilisation and efficiency improvements for systems that are not well utilised
  • Sharing options
    • Shared Everything
    • Shared Infrastructure
      • Virtual Machines
      • O/S virtualization
    • Minimal Sharing

Shared Everything

  • Resources are shared between all tenants or consumers
    • tenant: a service consumer
  • Common for the SaaS model
  • The application should provide tenant isolation
  • Data for multiple tenants is stored in the same database tables

Shared Infrastructure: Virtual Machines

  • Infrastructure shared via virtual machines
    • each tenant has its own virtual environment
    • Isolation provided by hypervisor
      • hypervisor: virtual machine manager, runs virtual machines
    • Resource contention depends on VM capability and configuration
    • Adds an additional layer and processes to run and manage

Shared Infrastructure: OS Virtualization

  • Infrastructure shared via OS Virtualization
    • Each tenant has its own processing zone
    • Isolation provided by the operating system
    • Resource contention depends on zone configuration
    • No VMs to run and manage, no abstraction layer between app & OS

Minimal Sharing

  • Maximized isolation between tenants' resources
  • Isolation provided by dedicated resource pools
  • Greatest degree of isolation, least economical

Elasticity

  • Def. (in physics): a physical property of materials which return to their original shape after they are deformed
  • "On-demand" expanding application resources across all its tiers
    • presentation, services, database, security, ...
    • application components can grow independently
  • Grow a tier without affecting, reconfiguring or changing the other tiers
  • Infrastructure as a Service elasticity
    • adding additional virtual machines, expanding infrastructure resources
    • adding computing resources will not have direct influence on the application elasticity
  • Platform as a Service elasticity
    • adding addtional application instances
  • Software as a Service elasticity
    • pricing elasticity - pay for chosen software package (basic, professional)

Scalability

  • Scaling Up
    • Number of users increases, need for more resources
    • Replication of app instances
  • See more details on sticky sessions in Lecture 8

Scalability

  • Scaling Down
    • Number of users decreases, no need for many resources
    • Deleting app instances

Scaling Down Issues

  • Session in a persistent storage
    • Stateful server
    • No sticky sessions – auto scaling
  • GAE implementation
    • New or updated session data
      → written to BigTable and the memcache
    • Reading the data
      • only from the cache, is fast
  • Limitations on data writing
    • GAE allows to update a single entity 5 times/sec.
    • When QPS > 5 (GAE scales up to 500 QPS)
      • the latency grows

Example Application Scalability

  • Response times × QPS for reading (R) and writing (W)
    • Writing scales well up to 5 QPS
    • Reading scales well for all QPS
Service Models

Web 2.0 Web Architecture

IaaS: Infrastructure as a Service

  • Provides fundamental computing resources and services for application providers
    • The consumer is able to deploy and run arbitrary software
    • Services for application providers
  • Infrastructure implications
    • Exposing of infrastructure resources through abstraction
    • Logical separation of underlying resources
    • Support for infrastructure isolation in multi-tenant environments
    • Support for infrastructure resources - compute (hardware/OS/VM), storage, network, etc.

IaaS: Infrastructure as a Service

  • Usage
    • Load predefined machine instances (micro, small, large, extra-large)
    • E.g., Linux OS, 613 MB of memory, 30 GB of Elastic Storage, Load Balancer, etc.
    • Pay-as-you-go - pay for the resources you use; no up-front costs
  • IaaS features
    • Elastic Storage
    • Monitoring resources (Amazon CloudWatch)
    • Auto Scalling of running instances
    • Elastic Load Balancing - distributing incoming
      traffic across multiple instances
  • IaaS providers
    • Amazon EC2, GoGrid, Rackspace,
      OpenNebula, ...

PaaS: Platform as a Service

  • Provides elastically scalable deployment environment for deployment of applications
    • No costs of buying and managing underlying infrastructure hardware and software
    • Services for application providers
  • Infrastructure implications
    • Scalable platform, deploy on-demand
    • Self service interface to deploy applications and services
    • Support for monitoring and measuring platform usage
    • Model supporting isolation in multi-tenant environments

PaaS: Platform as a Service

  • Usage
    • Choose software platform, e.g., JEE, .NET, Python, etc.
    • Pay-as-you-go - pay for the resources you use; no up-front costs
  • PaaS features
    • Auto Scalling and Load Balancing of applications
    • Persistent Storage - usually NoSQL database
    • Local development environment
    • Backends - for app instances with higher CPU
      and memory demants
    • Administration APIs for its services
  • PaaS providers
    • Google App Engine, Heroku,
      Windows Azure, etc.
  • Limitations
    • HTTP request deadline (30 - 60 sec)
    • No writes to file system, no thread support

SaaS: Software as a Service

  • Software delivery model for applications hosted in the cloud
    • typically software for end-users
    • services accessed using a thin web client
  • Infrastructure implications
    • Virtualization for multi-tenant, shared processes and services
    • Monitoring and measuring services for the customers
    • Protection and isolation of customer data in multi-tenant environment
    • Support for business processes, services and applications
  • SaaS providers
    • Google Apps, Salesforce, iCloud, Flickr, Picasa, ...

SaaS: Software as a Service

  • SaaS characteristics
    • Typically build on top of IaaS or PaaS
    • Configurable and customizable modern Web applications
    • Usually basic version for free, need to pay for pro version
    • Global availability - any computer, any device
    • Easy management - automatic and fast updates
    • Pay-as-you-go - pay for the time you use
Deployment Models

Overview

  • Four types of deployment models
    • Public Cloud
    • Private Cloud
    • Hybrid Cloud
    • Community Cloud

Public Cloud

  • Publicly accessible and self-service model
    • Access via Web services (SOAP, REST)
    • Access via management console
  • Infrastructure should ensure:
    • Security - it is a major concern!
    • Network Access
    • Contracts and SLA Management
    • Subscription Management
    • Multi-tenancy
    • Subscription Management
    • Metering, Billing, and Revenue Management
  • Its free or pay-per-use!
  • Applications, storage, and other resources available to the general public by a service provider
    • Google, Amazon, Microsoft, etc.

Private Cloud

  • Cloud infrastructure which operates for a single organization and is hosted internally
    • Emulated public cloud on internal resources
  • Implications
    • Internal cost allocation and chargeback
    • Self-managed scalability
    • Separation of Cloud resources from traditional IT resources
  • Benefits
    • Elasticity
    • Dynamic provisioning
    • Security?!

Hybrid Cloud

  • Composition of two or more public, private or community clouds
    • utilise the benefits of multiple deployment models
    • outsourcing non-critical functions and information
  • Infrastructure should ensure:
    • Inter-Cloud interaction
    • Synchronization Support between Clouds
    • Network Architecture: LB, Firewalls, VPN etc.
  • "Cloudbursting" - application runs in a private cloud and "bursts" into a public cloud when the demand for computing capacity is increased
    • For application with non-sensitive information
    • Cloudbursting applications in peak persiod, e.g. holiday shopping season

Community Cloud

  • Shared infrastructure between several organizations from a specific community
    • can be public or private cloud model
    • the community shares common concerns
    • costs spread over fewer users then a public cloud, and more then a private cloud
    • either on-premises or off-premises
  • Infrastructure should ensure:
    • Security
    • Inter-community economies of scale
    • Network Architecture: LB, Firewalls, VPN etc.
    • Compliance Control
  • Benefits
    • multi-tenancy
    • pay-as-you-go
Cloud Lifecycle Management

Cloud Projects Lifecycle

  • Managing the dynamic nature of the cloud environment

Plan and Setup

  • Cloud requirement analysis and cloud setup
  • Two types of cloud projects
    • Greenfield - start of a project with a new hardware, software and in some cases with new data center
    • Brownfield - transforming existing assets into a cloud environment
  • Setting up the Cloud
    • defining standardized service templates for databases and middleware platforms
    • create different types of services depending upon the business needs (IaaS, PaaS, etc.)
  • Cloud Infrastructure Administrator
    • set up servers, storage and database pools
  • Self-Service Administrator
    • defines access controls, quota, retirement policies and the charge plans with the service

Build, Test and Deploy

  • Packaging and Publishing application components to the Cloud as services
    • Components and applications published in the form of Assemblies
    • Production teams can deploy an application with few clicks
  • Testing an application before its deployment
    • capturing a production load and replaying it in a test environment
  • Application deployment in on-demand fashion
    • specifying the amount of underlying resources for each application component (CPU, memory, etc.)
    • define policies to scale out or scale back resources
    • Example: elastically scale out a web server if the processor load on existing web servers exceeds a certain threshold value

Monitor and Manage

  • Monitor health and performance of your Cloud
    • monitoring vast numbers of configurations continuously
    • discover changes
    • measure drifts
    • insight into system topologies
  • Template based monitoring setup
    • define monitoring settings, compliance standards and cloud policies
  • Managing SLAs between application owners and the cloud provider
    • defining management policies and automatically adjusting of the service resources

Meter, Charge and Optimize

  • Meter the usage and optionally chargeback
  • Different pricing models
    • CPU, memory and storage usage
    • application usage
    • database usage
    • middleware-level metrics
    • database usage
  • Extended pricing models
    • administrative expenses
    • energy utilization
    • combination of any two or more pricing models
  • Continuous optimization of resources
    • rediscovering assets
    • tuning capabilities in the operating system
    • tuning database
    • tuning middleware layers
